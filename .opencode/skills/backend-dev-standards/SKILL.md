# 后端开发规范

## API 设计

### 命名约定
- RESTful 风格：名词复数 + 资源层级
- `/api/users` `/api/users/:id/orders`
- 操作动作用 HTTP 方法：GET 查询、POST 创建、PUT 更新、DELETE 删除

### 响应格式
```json
{
  "code": 0,
  "data": {},
  "message": "ok"
}
```
- code=0 成功，非 0 为业务错误码
- 错误码全局唯一，有文档说明

### 参数校验
- 必填参数缺失返回 400 + 明确提示
- 参数类型错误返回 400
- 不信任客户端传来的任何数据，全部后端校验

## 数据库

### 操作规范
- 始终使用参数化查询，禁止字符串拼接 SQL
- 写操作使用事务，确保原子性
- 大批量操作分页处理，避免锁表

### 迁移
- Schema 变更通过迁移脚本管理，不手动改库
- 迁移脚本可回滚

## 错误处理

- 所有外部调用（数据库、第三方 API）包裹 try-catch
- 异常记录日志，向客户端返回通用错误信息（不暴露内部细节）
- 关键操作失败要有告警或重试机制

## 提交前自测

开发完成后，提交 Review 前必须确认：

- [ ] 验收标准全部通过
- [ ] API 所有端点可正常调用
- [ ] 参数校验覆盖：正常值、空值、超长、特殊字符
- [ ] 数据库操作正确，无 N+1 查询
- [ ] 错误场景有正确的错误码和提示
- [ ] 无硬编码的密钥/密码
- [ ] 代码已格式化

## 提交信息格式

```
feat(scope): 简短描述

- 改动点 1
- 改动点 2
```

## 注意事项

- 不在日志中输出密码、Token
- API 响应时间超过 1 秒考虑加缓存或优化
- 删除操作用软删除，不物理删除
